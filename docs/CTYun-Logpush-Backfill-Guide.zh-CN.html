天翼云 Logpush Backfill 指南

这是轻量版 backfill 方案。它在指定时间窗内把 R2 历史日志重新取出，过滤 Worker 子请求，转换为天翼云要求的格式，并按明确限速发送到客户接收端。

客户端约束

整个设计围绕三个客户端硬性要求：

1. 接收端吞吐上限（约 100,000 行/秒）

客户接收端无法承受任意速率。Sender 通过下式硬封顶：

max_concurrency × (1000ms / MIN_SENDER_INVOCATION_MS) × BATCH_SIZE
= 20 × (1000 / 200) × 1000
= 100,000 行/秒

这是理论上限，前提是每个 batch 接近 BATCH_SIZE = 1000 行，且客户 ACK 延迟低于 200ms。实际持续吞吐还受 Parser 上游供给限制 — 见下方 吞吐与调速。

2. 接收端不做去重

投递语义是 at-least-once（至少一次），不是 exactly-once。客户接收端无法去重，所以 Worker 通过两个 R2 标记（.done 与 .queued）尽量减少重复。常见场景（Parser 重试、Queue 重复投递）已经被消除，但还有两种残留场景无法消除：
  • fetch abort 但接收端已处理 — SEND_TIMEOUT_MS 触发，但 POST 实际已被接收端接受。Worker 无法得知。重试 → 接收端收到第 2 份。正常情况下概率 < 0.001%。
  • POST 成功但 .done 写入失败 — R2 PUT 永久性失败（已 3 次内部重试）。没有 .done 时下一次重试会重发。概率 < 0.0001%。
每个 batch 在 R2 key 中带稳定的 batch identity。能去重的接收端应按此 identity 去重。不能去重的接收端需要把残留重复率视为已知风险。

3. 不传 Worker 子请求

客户按父请求计费/审计。Parser 按以下规则在每条记录上过滤：

ParentRayID !== "00"        // 即子请求
WorkerSubrequest === true   // 显式子请求标记

两条规则都在格式转换之前应用。

与生产 Worker 共存

生产 worker ctyun-logpush-worker 实时把日志推到同一个客户接收端，量级相近。Backfill 在 Cloudflare 层完全隔离：

资源生产Backfill是否共享
Worker isolate / CPU 配额ctyun-logpushctyun-logpush-backfill独立
Queuesparse-queue / send-queueparse-queue-backfill / send-queue-backfill独立
R2 bucketcdn-logs-rawcdn-logs-raw共享（前缀不同）
R2 前缀processed/processed-backfill/<run-id>/独立
客户接收端共享共享共享（下游压力共担）

含义：backfill 假死（比如客户接收端卡住）不会消耗生产 worker 的 concurrency 或 queue 槽位。两个 worker 共享接收端带宽，所以 backfill 的显式限速是保护接收端的关键。

架构

R2 logs/
-> parse-queue-backfill
-> Parser  （gzip 解码 + 过滤 + 转换 + 切成 1000 行 batch）
-> processed-backfill/<run-id>/{batch}.txt + {batch}.queued
-> send-queue-backfill
-> Sender  （gzip + 签名 POST + .done 标记 + 清理）
-> 客户接收端
热路径里不使用 Durable Object。每个 raw .gz 文件独立解析并直接写成最终 batch 文件。每文件最后一个 batch 可能不满 1000 行 — 这是可接受的。

这套方案保证什么

  • 精确回放 [BACKFILL_START_TIME, BACKFILL_END_TIME] 内的日志
  • 只发送父请求：ParentRayID = "00" 且 WorkerSubrequest != true
  • 使用和生产一致的 CTYun 鉴权 secrets 做 POST
  • 通过 queue 并发 + 200ms invocation floor 对 sender 做显式限速
  • run 完成后自动清理临时 batch 文件（24 小时宽限）

部署

变量默认含义
BACKFILL_START_TIME""补传开始时间，ISO 8601
BACKFILL_END_TIME""补传结束时间，ISO 8601，必须 <= 当前时间
BACKFILL_ENABLED"false"必须显式 "true" 才运行

最大补传时间窗：7 天（MAX_RANGE_HOURS = 24 * 7）。更大时间窗需要拆成多次运行。

wrangler queues create parse-queue-backfill
wrangler queues create send-queue-backfill
wrangler queues create parse-dlq-backfill
wrangler queues create send-dlq-backfill

wrangler secret put CTYUN_ENDPOINT --config wrangler-backfill.toml
wrangler secret put CTYUN_PRIVATE_KEY --config wrangler-backfill.toml
wrangler secret put CTYUN_URI_EDGE --config wrangler-backfill.toml

wrangler deploy --config wrangler-backfill.toml

吞吐与调速

pipeline 有两个独立上限，取较低者：

阶段配置理论上限
Sendermax_concurrency=20 × 200ms floor × 1000 行/batch100,000 行/秒
Parsermax_concurrency=10 × 单 10 万行文件约 15 秒 wall time × 100 batch/文件~65,000 行/秒
默认配置下 Parser 才是真正瓶颈。持续吞吐约 65,000 行/秒，不是 Sender 的 100,000 名义上限。如果 65K 可接受，无需调参。

如果需要接近 100K，最稳妥的做法是调高 Parser max_concurrency：

目标持续吞吐Parser max_concurrencyBACKFILL_RATE（文件/分钟）风险
~65K 行/秒10（默认）60无 — 当前默认
~85K 行/秒1460低 — 仅多几个 Parser invocation
~100K 行/秒16–1860–80低 — 距 250 平台上限仍很远
不要调高 Sender max_concurrency 超过 20，不要缩短 MIN_SENDER_INVOCATION_MS 低于 200。两者都会超出接收端实测容量，并可能影响同一接收端上的生产 worker。

针对大 raw 文件，parse queue 特意配置为 max_batch_size = 1，这样每个大 gzip 文件独占一个 invocation，不会和另外两个大文件共享同一个 invocation 的 CPU 预算。

观察

curl https://ctyun-logpush-backfill.<your-subdomain>.workers.dev/backfill/status
wrangler tail ctyun-logpush-backfill

默认 /backfill/status 是"人话版"，关键字段：

字段含义
summary一句话状态总结
stage_codescanning_and_queueing / sending / sent_waiting_cleanup / cleaned
delivery_completedbatches_pending == 0 且全部 batch 已发送时为 true
fully_completed临时文件清理完才为 true
matched_raw_files时间窗内匹配的 raw 文件总数
batches_sent / batches_pending已发 / 待发 batch 数
log_lines_sent已发送到接收端的总行数

replay_window_beijing北京时间下的补传时间窗

同样这份"人话版"内容也会写进 cdn-logs-raw/backfill-state/status.json，方便直接在 R2 UI 查看。要排障看原始状态和底层 artifact 统计，用 GET /backfill/status?view=raw。

清理生命周期

所有 batch 发送完成后：

  1. inspectRunArtifacts — list 所有 batch artifact 并核实 .done 都存在
  2. cleanup.status = ready — 全部确认已发送后进入此状态
  3. 24 小时宽限期 — 临时文件保留供检查
  4. cleanup.status = deleting — 批量删除 processed-backfill/<run-id>/*
  5. cleanup.status = done — state 标记为 cleaned，run 彻底完成

宽限期是有意保留的：给运维方一个窗口在文件被删除前检查发送内容。

重跑同一时间窗

如果同一个 [A, B] 时间窗已经成功跑完，需要重新跑：

# 1. 删除 state 文件
wrangler r2 object delete cdn-logs-raw/backfill-state/progress.json
wrangler r2 object delete cdn-logs-raw/backfill-state/status.json

# 2. 重新部署，或等待下一次 cron（保持 BACKFILL_ENABLED="true"）

旧的 processed-backfill/<run-id>/ 前缀不会阻塞新的 run — 每次 run 用新的 run_id（带时间戳后缀）。

什么时候算结束

最稳妥的判断条件：

  • /backfill/status 显示 fully_completed = true
  • send-queue-backfill backlog 归零（在 Cloudflare Dashboard 可见）
