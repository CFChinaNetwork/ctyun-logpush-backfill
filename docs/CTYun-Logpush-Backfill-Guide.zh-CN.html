Cloudflare Worker Guide

天翼云历史日志补传指南

适用于客户漏收一段历史日志时，使用独立 Backfill Worker 从 R2 原始日志中补传指定时间窗口的场景。

概述 部署 观察 调速 清理 FAQ

概述

用途

精确回放指定时间窗 [A, B] 内的历史日志，不影响生产 CF 内部链路。

共享资源

Backfill 与生产共享同一个客户接收 endpoint，因此默认发送速率保持保守。

默认上限

默认发送上限为 <= 5,000 行/秒，实际速率取决于客户 ACK 时间与日志批次大小。

请求过滤

仅处理父请求 / top-level requests：ParentRayID = "00" 且 WorkerSubrequest != true。这是客户的特殊要求，用来排除 Workers 子请求。

自动清理

成功完成后会在一个较长的安全等待期后自动清理 processed-backfill/<run-id>/ 和发送统计文件，尽量不在客户 R2 留下尾巴。

数据流转如下：

R2 logs/ -> parse-queue-backfill -> Parser
        -> RunAggregator Durable Object
        -> processed-backfill/<run-id>/
        -> send-queue-backfill -> Sender -> customer endpoint

Backfill 只读取 logs/ 下的原始 Logpush 文件，不会改动客户原始日志。Parser 会逐条解压并过滤掉 Workers 子请求，只保留父请求。

部署

部署前，客户只需要确认这几个值：

字段含义
RUN_AGGREGATORDurable Object 绑定，用于跨多个 raw 文件聚合日志行并凑满 batch。
BACKFILL_START_TIME补传开始时间，ISO 8601 格式
BACKFILL_END_TIME补传结束时间，必须小于等于当前时间
BACKFILL_ENABLED默认建议保持 false，确认配置无误后再改成 true。
BACKFILL_RATE每分钟扫描多少个 raw 文件，当前示例值 20
SEND_TIMEOUT_MS等待客户 ACK 的最长时间，默认 300000
BATCH_SIZE每次 POST 最多带多少条日志，默认 1000

一次性初始化：

wrangler queues create parse-queue-backfill
wrangler queues create send-queue-backfill
wrangler queues create parse-dlq-backfill
wrangler queues create send-dlq-backfill

wrangler secret put CTYUN_ENDPOINT --config wrangler-backfill.toml
wrangler secret put CTYUN_PRIVATE_KEY --config wrangler-backfill.toml
wrangler secret put CTYUN_URI_EDGE --config wrangler-backfill.toml

部署：

wrangler deploy --config wrangler-backfill.toml

Durable Object 不需要额外手工创建，Wrangler 会按配置中的 migration 自动完成。

如果上一个 run 还没结束，请不要直接改时间窗重跑。必须等 /backfill/status 里的 status = "cleaned" 后，再改 BACKFILL_START_TIME / BACKFILL_END_TIME。

观察

最常用的两个观察入口：

curl https://ctyun-logpush-backfill.<your-subdomain>.workers.dev/backfill/status
wrangler tail ctyun-logpush-backfill
字段怎么理解
statusrunning 表示还在跑，done 表示 enqueue 阶段完成，cleaned 表示临时文件已自动清理。
phaseenqueue 表示还在扫 raw 文件，done 表示 raw 文件已经扫完。
enqueued_count命中的 raw .log.gz 文件数量，不是日志条数。
send_stats.success_count成功发给客户的 HTTP POST 次数。
send_stats.timeout_count等待客户 ACK 超时次数。
send_stats.ack_ms_avg/max客户接收端返回 ACK 的耗时。
send_stats.queue_wait_ms_avg/max消息在 send-queue-backfill 里排队等待被处理的时间。
queue_wait_ms 和 Queue 控制台里的 Average Consumer Lag Time 语义接近： 都是在看“消息进入队列后，等待被 consumer 处理的时间”。前者是 Worker 按单条消息记录的应用侧观测，后者是 Queue 原生的聚合平均值。
如果 ack_ms 高但 queue_wait_ms 低，说明主要是客户 ACK 慢；如果 queue_wait_ms 高但 ack_ms 不高，说明主要是 send queue 在堆积。

调速

默认发送速度上限的参数对应关系：

参数直白解释
BATCH_SIZE = 1000每次 POST 最多带 1000 条日志。
max_concurrency = 1同时最多只有 1 个 sender invocation 在跑。
max_batch_size = 1每个 sender invocation 只处理 1 个 send queue message，也就是大致只发 1 个 POST。
MIN_SENDER_INVOCATION_MS = 200每个 invocation 至少占用 200ms，对应每秒最多 5 次发送。
lines/s 上限
= max_concurrency × max_batch_size × (1000 / MIN_SENDER_INVOCATION_MS) × BATCH_SIZE
= 1 × 1 × 5 × 1000
= 5000 行/秒

实际速率通常会低于这个上限，因为客户 ACK 慢时，sender 会自然降速。

调速建议顺序：先看 SEND_TIMEOUT_MS 是否太短；再考虑增大 BATCH_SIZE；最后才考虑增加 sender 并发。不要指望提高 BACKFILL_RATE 来加快 drain，它只会让 backlog 增长更快。

清理

当满足以下条件时，Backfill 会自动清理临时文件：

  • status = done
  • processed-backfill/<run-id>/ 下已经没有待发送的 .txt 或 .queued 文件
  • 经过约 1 天的安全等待期

自动删除的内容：

  • processed-backfill/<run-id>/ 下的所有临时批次文件与 .done 标记
  • backfill-state/send-stats.json
为了防止同一配置在 Worker 还部署着时自动重跑，backfill-state/progress.json 会压缩成一个很小的完成标记并继续保留。等 Backfill Worker 不再需要时，删除 Worker 后再删这个小文件即可。

FAQ

Q: 这套 backfill 会影响生产吗？

A: 它不会共享生产的 Queue 和 processed 前缀，因此不会阻塞或污染生产的 CF 内部链路。但它仍然 POST 到同一个客户 endpoint，所以会增加一小部分补传流量。

Q: 怎么判断补传真的结束了？

A: 同时满足：/backfill/status 显示 status = done/cleaned，并且 send-queue-backfill backlog 归零。

Q: 如果客户 ACK 很慢怎么办？

A: 优先增大 SEND_TIMEOUT_MS，减少“客户其实收到了，但 ACK 回得慢，导致误 retry”的概率。
