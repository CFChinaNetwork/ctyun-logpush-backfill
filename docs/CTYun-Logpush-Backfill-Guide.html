CTYun Logpush Backfill Guide

This is the final lightweight backfill design. It replays historical R2 logs inside a precise time window, drops Worker subrequests, transforms each record to the CTYun partner format, and sends batches to the customer endpoint with explicit rate control.

Architecture

R2 logs/
-> parse-queue-backfill
-> Parser
-> processed-backfill/<run-id>/
-> send-queue-backfill
-> Sender
-> customer endpoint
No Durable Object is used in the hot path. Each raw file is parsed independently and written into final batch files directly.

What It Guarantees

  • Precise replay inside [BACKFILL_START_TIME, BACKFILL_END_TIME]
  • Only top-level requests are forwarded: ParentRayID = "00" and WorkerSubrequest != true
  • Authenticated POSTs using the same CTYun secrets as production
  • Sender-side rate cap through queue concurrency + a 200ms per-invocation floor
  • Temporary batch files are cleaned automatically after the run completes

Delivery Semantics

Important: this pipeline is at-least-once, not exactly-once. If a customer POST succeeds but the Worker cannot persist the .done marker afterward, the batch will be retried. If the receiver requires strict exactly-once behavior, dedupe by batch identity on the receiver side.

If you need to replay the exact same window again, delete cdn-logs-raw/backfill-state/progress.json and cdn-logs-raw/backfill-state/status.json first, then redeploy or wait for the next cron tick. Older processed-backfill/<run-id>/ prefixes do not block a fresh run.

Tuning

Current checked-in sender ceiling:

max_concurrency = 20
max_batch_size = 1
MIN_SENDER_INVOCATION_MS = 200
BATCH_SIZE = 1000

theoretical ceiling
= 20 * 1 * (1000 / 200) * 1000
= 100,000 lines/s

That ceiling assumes batches are close to 1000 lines and customer ACK stays under 200ms.

For large raw files, parse queue is intentionally configured as max_batch_size = 1 and max_concurrency = 10, so each large gzip file gets its own invocation instead of sharing CPU time with two other large files.

Monitoring

curl https://ctyun-logpush-backfill.<your-subdomain>.workers.dev/backfill/status
wrangler tail ctyun-logpush-backfill

The default /backfill/status response is human-friendly and includes:

  • summary: a plain-language one-line description
  • status_code and stage_code, each with its own explanation field
  • delivery_completed: whether all batches have been sent
  • fully_completed: whether the replay is truly finished, including cleanup
  • matched raw file count, sent batch count, sent log line count, pending counts, and batch-size distribution
  • Beijing-time timestamps only, plus a note explaining that the R2 Dashboard “Date Created” is not the right signal for the latest run state

The same human-friendly payload is also written to cdn-logs-raw/backfill-state/status.json so you can inspect it directly in the R2 UI. Use GET /backfill/status?view=raw if you want the raw state document plus low-level artifact stats.

Finished?

The safest completion rule is:

  • /backfill/status shows status = done or cleaned
  • send-queue-backfill backlog is zero
