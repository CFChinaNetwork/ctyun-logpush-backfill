CTYun Logpush Backfill Guide

Use this standalone Worker when the customer missed a historical slice of logs and needs a controlled replay from the original Logpush files stored in R2.

Overview Deploy Monitor Tuning Cleanup FAQ

Overview

Use case

Replay only the historical records inside a precise time window [A, B].

Isolation

The backfill Worker uses its own Worker name, Queues, and processed-backfill/<run-id>/ prefix.

Shared endpoint

The customer ingestion endpoint is still shared with production, so the default sender rate stays conservative.

Auto-cleanup

After a successful run, temporary batch artifacts are automatically removed from R2 after a safety delay.

Pipeline:

R2 logs/ -> parse-queue-backfill -> Parser -> processed-backfill/<run-id>/
        -> send-queue-backfill -> Sender -> customer endpoint

The Worker reads only from logs/. It does not modify the customer's original raw logs.

Deploy

Before deployment, make sure these values are set correctly:

FieldMeaning
BACKFILL_START_TIMERequired replay start time in ISO 8601.
BACKFILL_END_TIMERequired replay end time in ISO 8601 and must be <= now.
BACKFILL_RATEHow many raw files are scanned per cron minute. Current example: 20.
SEND_TIMEOUT_MSHow long the sender waits for the customer ACK before retrying.
BATCH_SIZEHow many log lines are packed into one HTTP POST at most.

One-time queue setup:

wrangler queues create parse-queue-backfill
wrangler queues create send-queue-backfill
wrangler queues create parse-dlq-backfill
wrangler queues create send-dlq-backfill

Set the three secrets:

wrangler secret put CTYUN_ENDPOINT --config wrangler-backfill.toml
wrangler secret put CTYUN_PRIVATE_KEY --config wrangler-backfill.toml
wrangler secret put CTYUN_URI_EDGE --config wrangler-backfill.toml

Deploy:

wrangler deploy --config wrangler-backfill.toml

Monitor

The two most useful commands are:

curl https://ctyun-logpush-backfill.<your-subdomain>.workers.dev/backfill/status
wrangler tail ctyun-logpush-backfill
FieldMeaning
statusrunning = in progress, done = enqueue finished, cleaned = temporary artifacts already auto-removed.
phaseenqueue while raw files are still being scanned, done when the raw file scan is finished.
enqueued_countNumber of raw .log.gz files matched in the requested window.
send_stats.success_countNumber of successful HTTP POSTs to the customer endpoint.
send_stats.timeout_countNumber of sender timeouts while waiting for ACK.
send_stats.ack_ms_avg/maxCustomer ACK latency from POST start to ACK received.
send_stats.queue_wait_ms_avg/maxHow long a send message sat in send-queue-backfill before a consumer picked it up.
queue_wait_ms vs Average Consumer Lag Time: the Queue dashboard metric is the queue-native aggregated average; queue_wait_ms is the Worker-side per-message approximation. They should generally point in the same direction.
High ack_ms with low queue_wait_ms means the customer endpoint is slow. High queue_wait_ms with low ack_ms means the sender queue is building backlog.

Tuning

Default throughput cap mapping:

ParameterPlain meaning
BATCH_SIZE = 1000Each HTTP POST can carry up to 1000 log lines.
max_concurrency = 1Only 1 sender invocation runs at the same time.
max_batch_size = 1Each sender invocation handles only 1 send-queue message, which is effectively one POST.
MIN_SENDER_INVOCATION_MS = 200Each sender invocation takes at least 200ms, which caps the sender at 5 sends per second.
lines/s cap
= max_concurrency × max_batch_size × (1000 / MIN_SENDER_INVOCATION_MS) × BATCH_SIZE
= 1 × 1 × 5 × 1000
= 5000 lines/s

Actual speed is often lower because the sender naturally slows down when the customer ACK is slow.

Recommended tuning order: first increase SEND_TIMEOUT_MS if the customer ACK is close to timing out; then increase BATCH_SIZE if the endpoint can accept larger POST bodies; only then consider increasing sender concurrency. Raising BACKFILL_RATE alone will only make backlog grow faster.

Cleanup

After a successful run, the Worker automatically removes the temporary artifacts after a safety delay of about 20 minutes:

  • All temporary artifacts under processed-backfill/<run-id>/
  • backfill-state/send-stats.json

The Worker keeps only a tiny backfill-state/progress.json completion marker while the Worker is still deployed. This prevents the same config from replaying again automatically.

Once the backfill Worker is no longer needed, delete the Worker and then remove the tiny progress marker if you want the bucket to contain only the customer's raw logs.

FAQ

Does backfill affect production?

It does not share production Queues or processed prefixes, so it cannot corrupt the production CF pipeline. It still POSTs to the same customer endpoint, so it adds controlled backfill traffic there.

How do I know backfill is truly finished?

The safest rule is: /backfill/status shows status = done/cleaned and the send-queue-backfill backlog is zero.

What if the customer ACK is slow?

Increase SEND_TIMEOUT_MS first. That reduces the chance of retrying a POST that the customer actually received but acknowledged late.
